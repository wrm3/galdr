# galdr-vault

the galdr vault is a structured folder of `.md` files that serves as persistent memory across sessions and projects. Content type is determined by frontmatter metadata. Project-scoped content (sessions, decisions) lives under `projects/{name}/`; everything else is global.

## Architecture Rule: Docker NEVER Touches the Vault

The vault is **local files only**. AI agents, hooks, and humans read and write vault files directly using native file operations on the host machine. The Docker MCP server has **zero vault access** — no bind mounts, no vault environment variables, no vault plugins.

| Component | Vault Access | How |
|-----------|-------------|-----|
| AI agents (Cursor, Claude, Gemini) | **YES** | Native file read/write |
| PowerShell hooks | **YES** | Native file read/write |
| Humans | **YES** | Any text editor, file explorer |
| Docker MCP server | **NO** | Returns data; agent writes to vault |
| Crawl4ai (host script) | **YES** | Runs on host; writes directly to vault |
| Video analyzer (Docker) | **NO** | Returns summary; agent writes to vault |

### Data Flow for Docker-Based Tools

```
1. Agent calls MCP tool (video_analyze, research_deep, etc.)
2. Docker processes the request, returns data to agent
3. Agent writes the result to the vault as a .md file
4. Agent calls vault_sync to index the note in the database
5. vault-reindex.ps1 updates _index.yaml (auto or manual)
```

## Vault Path Resolution

1. Read `.galdr/.vault_location` (a single-line file containing the absolute vault path)
2. If value is `{LOCAL}` or file is missing, use `.galdr/vault/` inside the current project (local mode)
3. If value is an absolute path, use that (shared vault accessible by all projects)
4. **Deprecated fallback**: if `.vault_location` doesn't exist, check `.env` for `GALDR_KNOWLEDGE_WELL_PATH=<path>` (backward compat only)
5. Local `.galdr/vault/` uses the **same folder structure** as the shared vault so content merges cleanly when the user later redirects to a shared path

To redirect to a shared vault, edit `.galdr/.vault_location` and replace `{LOCAL}` with the absolute path (e.g. `G:\galdr-vault`). The agent will offer to merge existing local vault content into the new location.

The `vault-resolve.ps1` hook exports: `$VaultPath`, `$ProjectId`, `$ProjectName`, `$ProjectDir`, `$SessionsDir`, `$DecisionsDir`.

## Folder Structure

```
{vault}/
  _index.yaml                     # Master catalog — auto-generated, do not edit
  projects/                       # Project-scoped content
    {project_name}/
      sessions/                   # Chat session summaries for this project
      decisions/                  # Architectural decisions for this project
  research/                       # Global research (not project-specific)
    platforms/                    # Crawled platform docs (crawl4ai)
      claude-code/
      cursor/
      gemini/
      openai/
      opencode/
    videos/                       # ALL video analyses
    papers/                       # Academic papers, whitepapers
    articles/                     # Web articles, blog posts
    harvests/                     # @g-harvest results
  knowledge/                      # Consolidated topic cards + cross-project corrections
```

### Key Design Principles

- **Project-scoped vs. global** — Sessions and decisions go under `projects/{name}/`. Research, videos, knowledge are global.
- **Same structure everywhere** — `.galdr/vault/` mirrors the shared vault exactly, enabling clean merges
- **No dual-write** — Each note lives in exactly one location
- **`_index.yaml` is the map** — Browsable catalog of all notes without needing database access

## Master Index (`_index.yaml`)

Auto-generated by `vault-reindex.ps1`. Contains every vault note with metadata for offline browsing:

```yaml
notes:
  - path: research/videos/2026-03-27_claude_code_memory_2_auto_dream_6pjETAf2XhU.md
    title: "Claude Code Memory 2.0 With UNLIMITED Memory!"
    type: video_analysis
    ingestion_type: video_analysis
    date: 2026-03-27
    topics: [AI, claude-code, memory, auto-dream]
    source: https://youtu.be/6pjETAf2XhU
    project_id: null
    refresh_policy: none
```

### At-Work Fallback (No Docker)

When MCP/Docker is unavailable, agents can:
1. Read `_index.yaml` to find relevant notes by topic/type/date
2. Read the actual `.md` files directly
3. Use `grep` or file search on vault contents

No database or MCP tools required — pure file operations.

## Note Format

Every vault note MUST have YAML frontmatter. Required fields: `date`, `type`, `ingestion_type`, `source`, `title`, `topics`. Optional: `project_id`, `refresh_policy`, `refresh_after`, `expires_after`, `source_volatility`, `source_notes`.

```yaml
---
date: 2026-03-27
type: video_analysis
ingestion_type: video_analysis
source: https://youtu.be/xyz
title: "Descriptive Title"
topics: [AI, claude-code, memory]
project_id: null
refresh_policy: none
source_volatility: low
---
```

### Quick Reference

**`type`** (determines folder): `video_analysis`, `platform_doc`, `article`, `paper`, `harvest`, `session`, `decision`, `correction`, `preference`, `knowledge_card`

**`ingestion_type`**: `video_analysis`, `crawl4ai`, `url_fetch`, `manual`, `harvest`, `session`, `research`, `paper`, `decision`, `correction`, `knowledge_card`

**`topics`**: Multi-valued descriptive labels. Use multiple per note. Think about what an AI or human would search for.

**`refresh_policy`**: `none` | `daily` | `weekly` | `monthly` | `on_demand`

**`source_volatility`**: `low` (permanent content) | `medium` (changes occasionally) | `high` (changes frequently)

Full type/status/tag/field reference with folder mappings and expiration policies: see `reference/vault_enums.md`

## Writing to the Vault

### Video Analysis
```
1. Call video_analyze MCP tool
2. Write result to: {vault}/research/videos/{date}_{slug}_{video_id}.md
3. Call vault_sync with the relative path and content
```

### Platform Docs (crawl4ai)
```
1. Run: python scripts/platform_crawl.py --target {name}
2. Script writes directly to .platforms/ AND vault/research/platforms/
3. Files land in: {vault}/research/platforms/{platform_name}/
```

### Web Articles / URLs
```
1. Fetch URL content
2. Write to: {vault}/research/articles/{date}_{slug}.md
3. Call vault_sync
```

### Harvest Results
```
1. Analyze external source
2. Write to: {vault}/research/harvests/{date}_{slug}.md
3. Call vault_sync
```

### Session Summaries
```
1. agent-complete.ps1 auto-writes to: {vault}/projects/{project_name}/sessions/{date}_{time}_{id}.md
2. Hook auto-calls vault-sync-to-db.ps1
```

### Decisions
```
1. Write to: {vault}/projects/{project_name}/decisions/{date}_{slug}.md
2. Call vault_sync
```

### Knowledge Cards (Consolidated)
```
1. Synthesize from multiple source notes
2. Write to: {vault}/knowledge/{topic_slug}.md
3. Link back to source notes via [[wikilinks]]
4. Call vault_sync
```

## Searching the Vault

### With Docker/MCP (preferred)
- `vault_search_all` — Semantic search across vault_notes, platform_docs, and agent_memory
- `vault_search` — Search vault_notes only, with type/topic filters
- `vault_list` — List notes by path prefix

### Without Docker (at-work fallback)
- Read `_index.yaml` for the full catalog
- Filter by topics, type, date in the YAML
- Read individual `.md` files directly

## Regenerating the Index

Run manually or let hooks call it:
```powershell
powershell -ExecutionPolicy Bypass -File .cursor/hooks/vault-reindex.ps1
```

The index is also regenerated by:
- `agent-complete.ps1` after writing session stubs
- `vault-bridge-platforms.ps1` after copying platform docs
- `g-cleanup` skill during nightly maintenance

## Knowledge Cards

Knowledge cards are synthesized notes in `knowledge/` that consolidate information from multiple source notes into a single authoritative reference on a topic.

### When to Create
- A topic has 3+ source notes (videos, articles, papers)
- User explicitly asks to consolidate knowledge on a topic
- The `g-knowledge-refresh` skill identifies convergent sources

### Format
```yaml
---
date: 2026-03-28
type: knowledge_card
ingestion_type: knowledge_card
title: "Topic: Memory Systems in AI Agents"
topics: [AI, memory, agents, RAG, knowledge-graph]
source_notes:
  - research/videos/2026-03-27_claude_code_memory_2.md
  - research/videos/2026-03-27_7_levels_claude_code.md
  - research/platforms/claude-code/memory-docs.md
refresh_policy: on_demand
source_volatility: high
expires_after: 2026-06-28
---

# Memory Systems in AI Agents

## Summary
{Synthesized overview from all sources}

## Key Concepts
{Merged and deduplicated key points}

## Sources

- [[research/videos/2026-03-27_7_levels_claude_code.md]] — 7 Levels of Claude Code
```
