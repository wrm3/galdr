# g-kpi

Track and analyze agent performance metrics after every heartbeat routine run.

## When to Use

- After a heartbeat routine completes (automatic integration)
- To view KPI trends for a routine (`@g-kpi report <routine>`)
- To add a human observation (`@g-kpi observe <routine> "<note>"`)
- To check baselines and anomalies (`@g-kpi baselines`)

## KPI Capture (Post-Run)

After every heartbeat run, append a JSONL entry to `.galdr/logs/kpi/{date}_{routine}.jsonl`:

```json
{
  "run_id": "hb-{date}-{time}-{routine}",
  "routine": "{routine_name}",
  "agent": "{agent_name}",
  "adapter": "{adapter}",
  "started": "{ISO8601}",
  "finished": "{ISO8601}",
  "duration_seconds": 0,
  "status": "success|failed|skipped",
  "cost_estimate_usd": 0.00,
  "metrics": {},
  "errors": 0,
  "warnings": 0
}
```

The `metrics` object contains routine-specific KPIs as defined in `.galdr/KPI_DEFINITIONS.md`.

## Baseline Calculation

After 5+ runs of a routine, compute rolling baselines stored in `.galdr/logs/kpi/baselines.json`:

```json
{
  "routine_name": {
    "baseline_window": 14,
    "sample_count": 10,
    "baselines": {
      "duration_seconds": { "mean": 0, "stddev": 0, "p95": 0 },
      "cost_estimate_usd": { "mean": 0, "stddev": 0, "p95": 0 }
    },
    "last_updated": "YYYY-MM-DD"
  }
}
```

Regenerated by `g-cleanup` nightly or on-demand via `@g-kpi baselines`.

## Anomaly Detection

Compare each run's KPIs against baselines:

| Condition | Action |
|-----------|--------|
| Value > mean + 2*stddev | Log WARNING in heartbeat run report |
| Value > mean + 3*stddev | Add to WAKEUP_QUEUE.md for user attention |
| 3 consecutive runs trending worse | Flag in next SWOT review |
| Matches KPI_DEFINITIONS critical threshold | Immediate WAKEUP_QUEUE entry |

## Human Observations

Append to `.galdr/logs/kpi/observations.jsonl`:

```json
{
  "date": "YYYY-MM-DD",
  "routine": "routine_name",
  "observer": "user",
  "type": "correction|praise|concern|suggestion",
  "content": "Free-text observation",
  "action": "Suggested follow-up action"
}
```

## Commands

- `@g-kpi report <routine>` — Show trend summary for a routine (last 14 days)
- `@g-kpi observe <routine> "<text>"` — Record a human observation
- `@g-kpi baselines` — Recalculate all baselines from JSONL data
- `@g-kpi anomalies` — List recent anomalies across all routines

## Self-Improvement Loop

```
Run routine -> Capture KPIs -> Compare to baseline
                                    |
                    +----------------+----------------+
                    v                v                 v
              Normal range     Warning zone      Critical zone
              (log only)       (flag in report)  (wakeup + alert)
                                    |                 |
                                    +--------+--------+
                                             v
                                    SWOT review picks up
                                    trends and suggests
                                    skill/config changes
                                             |
                                             v
                                    User approves changes
                                    -> skills updated
                                    -> next run uses new config
                                    -> KPIs measured again
```

## File Locations

| File | Purpose |
|------|---------|
| `.galdr/KPI_DEFINITIONS.md` | Metric definitions with thresholds |
| `.galdr/logs/kpi/{date}_{routine}.jsonl` | Per-run KPI entries (append-only) |
| `.galdr/logs/kpi/baselines.json` | Rolling baseline calculations |
| `.galdr/logs/kpi/observations.jsonl` | Human feedback entries |
